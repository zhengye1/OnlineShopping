## 1. Goal

Add a production-quality React frontend to the existing Spring Boot e-commerce backend, delivered as a **course-driven portfolio project**. Two simultaneous outcomes:

1. **Curriculum** — 25 frontend lessons + 1 backend bridge lesson, following the same `核心概念 + 面試答法 + practical deliverable` pattern as Lessons 01-24.
2. **Portfolio** — a deployable full-stack e-commerce SPA on AWS, demonstrating FAANG-level frontend competency (RSC, SSR, hydration, Core Web Vitals, accessibility, testing).

## 2. Non-goals

- **Not a pure tutorial replica of popular courses** — content is driven by this concrete e-commerce domain.
- **Not a rewrite of the Spring Boot backend** — backend changes are strictly additive (CORS config, 1-2 BFF-friendly endpoints, OpenAPI spec generation).
- **Not a multi-tenant / SaaS product** — single-instance e-commerce app sufficient for learning + portfolio.
- **PWA, WebSockets, i18n, Framer Motion animations are out of scope** for core curriculum (reserved as optional bonus lessons).

## 3. Scope summary

| Dimension | Decision |
|---|---|
| Framework | Next.js 15 (App Router, RSC, Server Actions) + React 19 + TypeScript 5.7 strict |
| Styling | Tailwind CSS 4 + Shadcn/ui (Radix primitives) + Lucide icons |
| Client state | Zustand (primary) — Redux Toolkit covered in dedicated comparison lesson |
| Server state | Tanstack Query v5 on client; native `fetch` + cache tags on server |
| Forms | React Hook Form + Zod |
| API typing | OpenAPI spec generated by Spring Boot (`springdoc-openapi`) → consumed by `openapi-typescript` |
| Testing | Vitest, Testing Library, MSW, Playwright, Storybook 8 |
| Quality | ESLint + Prettier + strict TS + Lighthouse CI + Web Vitals reporting |
| Package manager | pnpm |
| Runtime | Node 20 LTS |
| Repo layout | Monorepo — frontend lives in `web/` inside existing `OnlineShopping/` repo |
| BFF | Next.js server acts as BFF; browser never calls Spring Boot directly |
| Auth token | JWT in HttpOnly cookie, never in localStorage |
| Local dev | Extended `docker-compose.yml` (adds `web` service with hot-reload volume mount) |
| Prod deploy | AWS ECS — new service `online-shopping-web` alongside existing backend, CloudFront in front |

## 4. Architecture

### 4.1 Runtime topology

```
Browser
   │ HTTPS
   ▼
CloudFront CDN
   │
   ├── /_next/static/*   → S3 (1 year immutable cache)
   │
   ▼
Application Load Balancer
   │
   ├── /api/v1/*   → ECS service: online-shopping       (existing Spring Boot)
   │                   │
   │                   └── Redis / MySQL / Elasticsearch / RocketMQ
   │
   └── /*          → ECS service: online-shopping-web   (new Next.js)
                       │ internal
                       └──→ backend.internal:8080 (Spring Boot)
```

### 4.2 Request patterns

Three distinct data-fetching patterns, each used intentionally:

| Pattern | Where | When | Example |
|---|---|---|---|
| RSC `fetch` | Server component | SEO-critical initial render | Product detail page, home feed |
| Tanstack Query | Client component | Interactive, paginated, refetch | Search results, order history |
| Server Action | `<form action={...}>` | Mutations | Add to cart, checkout, profile update |

### 4.3 Auth flow

```
Login form submit (Server Action)
  └─► Spring Boot /api/v1/auth/login
        └─ JWT returned
             └─ Next.js server sets HttpOnly cookie, redirects

Subsequent requests
  Browser cookie auto-sent → Next.js middleware
    ├─ extract JWT from cookie
    ├─ if protected route + missing cookie → redirect /login?from=…
    └─ proxy to Spring Boot with Authorization: Bearer <jwt>
```

- JWT is **only** in HttpOnly cookie (XSS-hardened).
- CSRF: Server Actions include built-in origin check; custom POST endpoints use double-submit cookie pattern.

### 4.4 Error handling (4 layers)

1. **Network** — typed API client throws `ApiError { status, message, traceId }` matching Spring Boot `@ControllerAdvice`.
2. **Route** — `app/**/error.tsx` catches unhandled errors per route segment.
3. **Global** — `app/global-error.tsx` catches root-level crashes.
4. **Component** — `<ErrorBoundary>` around risky widgets (payment form, embeds).

User-facing behavior:
- 400 (validation) → inline form error via Zod message
- 401 → auto redirect to `/login?from=<current>`
- 403 → "permission denied" page
- 404 → route-local `not-found.tsx`
- 5xx → generic error page with retry + `traceId` (for support)

All errors forwarded to `/api/v1/metrics/errors` (reuses Lesson 20 monitoring infrastructure).

## 6. Backend changes (minimal, additive)

Taught in **Lesson 25 (backend bridge)**. Strictly no breaking changes to existing endpoints.

| Change | File | Purpose |
|---|---|---|
| CORS config | `config/CorsConfig.java` | Allow `web` service origin (dev) + prod domain |
| Aggregate feed endpoint | `controller/FeedController.java` + `/api/v1/feed` | Home page: featured + categories + promos in 1 call (avoid 6-request waterfall) |
| Web Vitals ingest | `controller/MetricsController.java` + `/api/v1/metrics/vitals` | Receive LCP/INP/CLS from browser → Prometheus (reuse Lesson 20) |
| Error ingest | `/api/v1/metrics/errors` | Receive frontend exceptions → Prometheus |
| OpenAPI spec | `pom.xml` add `springdoc-openapi-starter-webmvc-ui` | Generate `/v3/api-docs` for frontend codegen |

Continues numbering from L24. Each lesson follows the existing `核心概念 + 面試答法 + practical deliverable` pattern.

## 8. Testing strategy

**Pyramid:**

```
          ┌─────────────────┐
          │  E2E (3-5)      │  Playwright
          │  critical flows │  login → browse → cart → checkout → order
          ├─────────────────┤
          │  Integration    │  Testing Library + MSW
          │  (page-level)   │  render page with mocked network, assert UI
          ├─────────────────┤
          │  Unit (most)    │  Vitest
          │                 │  hooks, utils, Zod schemas, Zustand stores
          ├─────────────────┤
          │  Visual         │  Storybook + visual diff
          │                 │  every component × every state (loading / populated / empty / error)
          └─────────────────┘
```

**Coverage target: 80%+** (matches global CLAUDE.md rule).

**TDD discipline per lesson:** test first (RED) → implementation (GREEN) → refactor. Aligns with `tdd-guide` agent and user's global TDD rule.

## 9. CI/CD

New workflow: `.github/workflows/web.yml`

```
trigger: push with paths: web/**

jobs:
  lint       → eslint + prettier --check + tsc --noEmit
  test       → vitest --coverage (fail if < 80%)
  e2e        → playwright (against docker-compose backend)
  build      → next build + bundle-size check
  lighthouse → performance budget (LCP < 2.5s, INP < 200ms, CLS < 0.1)
  docker     → multi-arch build, push to ECR
  deploy     → ECS service update (main branch only)
```

Backend workflow unchanged. Path filters prevent cross-triggering.

## 10. Deployment

### 10.1 Local dev

```
docker-compose -f docker-compose.yml -f docker-compose.dev.yml up
```

Starts mysql, redis, elasticsearch, rocketmq, backend (Spring Boot), **and web (Next.js with hot reload)**.

- Backend: `http://localhost:8080`
- Web: `http://localhost:3000`
- Web-to-backend via docker network: `http://backend:8080`

### 10.2 Production (AWS, taught in L49)

Extends Lesson 19 ECS stack:

```
Route 53 (shop.yourdomain.com)
    │
    ▼
CloudFront
    │
    ├── /_next/static/*  → S3 (1 year immutable)
    │
    ▼
Application Load Balancer (shared VPC with backend)
    │
    ├── /api/v1/*  → ECS service: online-shopping       (existing)
    └── /*         → ECS service: online-shopping-web   (new — Next.js SSR)
```

- Shared VPC, shared cluster, two ECS services
- Web container calls backend internally via service discovery (`backend.internal:8080`)
- CloudFront caches static assets + ISR HTML
- WAF rules: rate limiting, optional geo restriction

## 13. Risks + mitigations

| Risk | Mitigation |
|---|---|
| Next.js 15 + React 19 breaking changes mid-course | Pin versions in `package.json`; bump intentionally between phases |
| API drift between Java DTOs and TS types | OpenAPI codegen in CI; build fails on mismatch |
| Coupling between backend and frontend lesson cadence | Bridge lesson (L25) frontloaded; frontend lessons stay independent afterward |
| Stripe test account / payment setup friction | Include setup script + `.env.example`; optional Stripe-sandbox-only mode for L39 |
| ECS cost during learning | Teach pause/resume of ECS tasks; optional local-only mode (skip L49 deployment until ready) |
| Course taking longer than 15 weeks | Phases are independently valuable — if time-constrained, stop after L43 (features complete) still yields portfolio + most interview prep |
